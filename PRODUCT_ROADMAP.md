# MicroClaw 功能清单与三期落地路线图

更新时间：2026-03-15

## 1. 文档目的

这份文档用于回答两个问题：

1. 现在做的内容是否符合逻辑。
2. 一期/二期/三期到底做什么、如何落地、如何验收。

## 2. 当前状态（已完成）

当前代码属于 **Phase 0：基础骨架**，目标是把工程跑起来并打通前后端。

- Gateway：健康检查、登录占位、SignalR 占位
- WebUI：路由与基础页面（Dashboard/Login）
- 联通：前端已可调用网关健康检查与登录占位接口

结论：当前实现逻辑正确，属于正式业务功能前的必要准备阶段。

## 2.1 技术架构（当前与目标）

### 2.1.1 分层架构图（目标态）

1. 接入层
   - WebUI（Vue 3）
   - Channels（飞书 / 企业微信 / 微信）

2. 网关层（Gateway, .NET 10）
   - REST API（配置、管理、查询）
   - SignalR（实时事件、日志流）
   - Auth + RBAC + 审计

3. 领域服务层
   - DNA 记忆服务
   - 情绪引擎
   - 表观遗传服务
   - 认知整理调度器（Cron）
   - 会话与消息路由

4. 模型抽象层（Model Provider Layer）
   - IModelProvider（统一契约）
   - OpenAIProvider / ClaudeProvider
   - ModelRouter（选型、回退、熔断）

5. 基础设施层
   - PostgreSQL/SQLite（元数据）
   - Markdown 文件系统（基因文件）
   - Redis（缓存/队列，可二期）
   - Logging + Metrics（可观测）

### 2.1.2 运行拓扑（部署视角）

- webui 容器：提供管理界面
- gateway 容器：统一 API、WS、渠道回调、模型路由
- db 容器：持久化元数据
- 可选 redis 容器：重试队列、事件缓存
- 外部 Provider：OpenAI、Claude
- 外部 Channels：飞书、企业微信、微信

### 2.1.3 当前进度映射（你现在看到的代码）

- 已落地：网关层基础骨架（health/login 占位/signalr 占位）
- 已落地：接入层基础骨架（webui 路由与页面）
- 待落地：领域服务层、模型抽象层、渠道适配层

## 3. 产品核心能力域（总清单）

### A. DNA 记忆系统

1. Markdown 基因文件 CRUD（IDENTITY / SOUL / USER / MEMORY）
2. 记忆提炼 / 模糊 / 遗忘规则配置
3. 记忆版本回溯与差异对比
4. Markdown 导入 / 导出

价值：构建可演化认知结构，保留“类人记忆”特征。

### B. 情绪数值体系

1. 核心情绪值（警觉度 / 心情 / 好奇心 / 信心）实时监控
2. 情绪触发规则配置
3. 情绪驱动行为模式（探索 / 谨慎 / 休息）
4. 情绪曲线可视化

价值：让行为决策具备稳定人格倾向，而非固定模板响应。

### C. 表观遗传机制

1. 基因表达开关（按环境触发）
2. 甲基化标记管理（记忆/行为权重）
3. 基因表达效果预览

价值：DNA 结构稳定，行为模式动态适应。

### D. 定时认知整理

1. 记忆碎片整合（对话定时提炼）
2. 做梦模式（离线总结规律）
3. 定时任务配置（Cron）

价值：实现主动学习与持续进化。

### E. 渠道接入系统（Channels）

1. 渠道适配器（飞书 / 企业微信 / 微信）
2. Webhook 回调接入、签名验签、重放防护
3. 统一消息模型（收消息 / 发消息 / 回执 / 重试）
4. 渠道路由策略（渠道账号 -> Agent -> Session）
5. 渠道配置管理与连通性检测

价值：让 Agent 能在真实业务沟通渠道中运行，形成可用的生产入口层。

### F. 模型 API 抽象层（LLM Providers）

1. 统一模型调用接口（文本、流式输出、工具调用）
2. Provider 适配器（OpenAI / Claude，后续可扩展）
3. 模型路由策略（按 Agent、任务类型、成本、延迟）
4. 失败回退机制（主模型失败后自动切换）
5. Provider 配置管理（Key、BaseUrl、模型白名单）

价值：避免业务层与单一模型厂商耦合，支持持续扩展和低成本切换。

### G. 工具编排与 MCP 服务层（新增）

1. RIBOSOME 工具定义外部化（JSON 配置驱动）
2. MCP Server 动态工具注册与执行路由
3. Tool 输入 Schema 统一校验与版本管理
4. Tool 能力分级（核心工具/扩展工具）
5. 工具加载优先级（用户目录 -> 模板目录 -> 内置默认）

价值：将“工具定义”与“业务代码”解耦，支持快速扩展与低风险演进。

### H. 安全与避害系统（Nociception，新增）

1. 痛觉记忆模型（触发点/后果/规避策略/严重度）
2. 危险操作前置检查（执行前预警与拦截）
3. 痛觉与情绪联动（受伤后进入谨慎模式）
4. 风险操作白名单/灰名单策略

价值：把“安全经验”沉淀为可复用规则，降低高风险误操作概率。

### I. 向量记忆与 RAG 检索（新增）

1. 向量存储接口（可替换内存版/持久化版）
2. 嵌入服务抽象（默认零外部依赖实现）
3. 语义检索 + 关键词混合检索
4. 长文本分块、索引重建、检索统计
5. 记忆来源分层索引（长期记忆/日志/技能/实体）

价值：让“记忆召回”从关键词匹配升级到语义相关，提高上下文命中率。

### J. Heartbeat 自主执行（新增）

1. Heartbeat 任务模型与任务运行器
2. AI CLI 检测与执行适配层
3. 定时自主任务（巡检/整理/反思/修复）
4. 任务成功率、失败重试与告警

价值：让系统具备低人工干预的持续运行能力。

## 4. 分期规划（一期/二期/三期）

## 一期（MVP，4~6 周）

目标：先跑通“可用闭环”，服务单租户自托管。

### 一期功能

1. 网关与权限基础
   - JWT 登录鉴权
   - RBAC（admin/user）
   - 基础审计日志

2. DNA 记忆系统（MVP 子集）
   - 基因文件 CRUD（IDENTITY/SOUL/USER/MEMORY）
   - 文件版本快照（保存历史）
   - 导入/导出 Markdown

3. 定时认知整理（MVP 子集）
   - Cron 任务配置
   - 基础提炼任务：对话 -> memory 摘要

4. WebUI 控制台
   - 登录
   - 基因文件管理页
   - 渠道配置管理页（飞书/企业微信）
   - 渠道连通性测试页
   - 定时任务配置页
   - 运行日志页

5. 渠道接入（MVP 子集）
   - 飞书机器人接入（事件回调 + 消息收发）
   - 企业微信应用接入（事件回调 + 消息收发）
   - 统一渠道消息入站接口（用于路由到 Agent）

6. 模型 API 抽象（MVP 子集）
   - 定义统一模型接口（Chat / Stream）
   - OpenAI Provider 适配器
   - Claude Provider 适配器
   - WebUI 可切换 Agent 默认模型

7. 工具编排与网关路由（MVP 子集，新增）
   - 网关内消息总线与渠道管理器（ChannelManager + MessageBus）
   - 统一入站消息信封（channel/user/session/payload）
   - 最小 MCP 服务（健康检查 + 工具列表占位）
   - RIBOSOME 配置文件模板与加载器最小实现

### 一期验收标准

- 用户可登录并基于角色访问页面
- 4 类基因文件可增删改查并保留历史版本
- 可配置并执行定时整理任务
- 飞书与企业微信可完成双向消息打通（收/发）
- 同一会话可在 OpenAI 与 Claude 间切换且不改业务代码
- 日志可追踪“谁在何时修改了什么”
- 渠道消息通过统一消息信封进入网关路由（可追踪 session）
- MCP 可返回最小工具清单，RIBOSOME 模板可被加载

---

## 二期（增强，6~10 周）

目标：让系统具备“行为驱动能力”。

### 二期功能

1. 情绪数值体系
   - 四维情绪实时值
   - 规则引擎（事件触发增减）
   - 行为模式切换（探索/谨慎/休息）
   - 情绪时序图

2. DNA 记忆增强
   - 记忆模糊与遗忘规则（时间衰减、权重阈值）
   - 版本差异对比 UI（diff 视图）

3. 定时认知整理增强
   - 做梦模式（离线聚类/归纳）
   - 手动触发与自动触发并存

4. 渠道接入增强
   - 微信接入（需合规与方案评估后落地）
   - 多渠道统一路由（会话连续性）
   - 重试队列与死信处理

5. 模型 API 抽象增强
   - 增加更多 Provider（如 Azure OpenAI / OpenRouter）
   - 引入模型路由策略（质量优先 / 成本优先）
   - 引入 Token/成本统计与预算保护

6. 工具与记忆增强（新增）
   - MCP 动态工具注册（由 RIBOSOME 配置驱动）
   - 痛觉记忆系统（危险操作预警 + 拦截）
   - 向量记忆与混合检索（RAG）
   - 好奇心目标引擎（按情绪状态生成探索目标）

### 二期验收标准

- 情绪值会随事件变化并可回放曲线
- 行为模式对响应策略产生可见影响
- 遗忘规则生效，低价值记忆会被模糊或降权
- 多渠道路由稳定，异常消息可追踪与重放
- 模型路由策略生效，调用质量与成本可观测
- 痛觉规则可阻断高风险操作并给出规避建议
- RAG 可返回语义相关上下文，召回结果可观测

---

## 三期（进化，10~16 周）

目标：实现“动态表达”的完整认知系统。

### 三期功能

1. 表观遗传机制完整化
   - 基因表达开关与环境触发
   - 甲基化标记管理
   - 表达效果预览（变更前后模拟）

2. 多 Agent 协同（可选）
   - 基因模板复用
   - Agent 之间记忆隔离与共享策略

3. 运维与可观测性增强
   - 指标面板（任务成功率、记忆增长、情绪波动）
   - 告警机制（任务失败、规则异常）

4. 渠道策略编排（进化）
   - 按用户/场景/时间自动切换渠道策略
   - 渠道健康评分与自动降级
   - 跨渠道会话连续性优化

5. 模型编排与智能路由（进化）
   - 多模型协同（推理模型 + 执行模型）
   - 自动回退与熔断
   - 任务级模型选择（总结/代码/对话分别选型）

6. 自主执行闭环（新增）
   - Heartbeat 自主任务编排与执行器
   - AI CLI 能力探测与策略降级
   - 自主任务审计、失败恢复与安全限流

### 三期验收标准

- 环境变化可触发表观遗传规则
- 表达预览与实际行为变化一致
- 系统具备可观测、可回溯、可调优能力
- 渠道故障可自动降级，不影响核心会话服务
- 模型故障可自动切换，核心任务成功率保持稳定
- Heartbeat 任务可稳定自主执行并可审计可回放

## 5. 模块落地顺序（为什么这样做）

推荐顺序：

1. DNA 记忆 CRUD + 版本管理（数据底座）
2. 定时认知整理（持续输入/加工数据）
3. 情绪数值体系（使用整理后的信息驱动行为）
4. 表观遗传机制（在稳定数据与行为系统上做动态表达）

原因：后面的模块都依赖前面的数据与规则基础，逆序开发会导致大量返工。

## 6. 技术架构与落地建议（与你当前技术栈对齐）

> 说明：本章节是“目标架构细化”，第 2.1 节是“快速总览”。

### 总体架构（逻辑分层）

1. 接入层
   - WebUI（Vue 3）
   - Channels（飞书 / 企业微信 / 微信）

2. 网关层（Gateway）
   - REST API（配置、管理、查询）
   - WebSocket/SignalR（实时事件、日志流）
   - Auth + RBAC + 审计

3. 领域服务层
   - DNA 记忆服务
   - 情绪引擎
   - 表观遗传服务
   - 认知整理调度器（Cron）
   - 会话与消息路由

4. 模型抽象层（Model Provider Layer）
   - 统一接口：IModelProvider
   - Provider 适配器：OpenAIProvider、ClaudeProvider
   - ProviderRegistry（注册与发现）
   - ModelRouter（策略选型与回退）

5. 基础设施层
   - 元数据存储（PostgreSQL/SQLite）
   - 基因文件存储（Markdown 文件系统）
   - 缓存与队列（Redis，可二期）
   - 日志与指标（结构化日志 + 监控）

### Gateway（.NET 10）

- API：Minimal API + SignalR
- 鉴权：JWT Bearer + Policy-based RBAC
- 任务：Quartz.NET（Cron）
- 渠道：Adapter 模式（Feishu / WeCom / WeChat Connector）
- 模型：Provider 抽象层（OpenAI / Claude 起步，后续扩展）
- 存储建议：
  - 元数据：PostgreSQL（或一期先 SQLite）
  - 基因文件：磁盘 Markdown（按 Agent 隔离目录）
  - 缓存：Redis（可二期再上）

### WebUI（Vue 3）

- 状态管理：Pinia
- 路由：Vue Router
- 图表：ECharts（情绪曲线）
- Diff：Monaco Diff 或轻量 Markdown diff 组件
- 渠道管理：渠道配置表单 + Webhook 状态面板
- 模型管理：Provider 配置页 + Agent 模型策略页

### 模型 API 抽象设计建议

1. 统一请求/响应 DTO
   - ModelRequest（messages、temperature、maxTokens、tools、stream）
   - ModelResponse（text、toolCalls、usage、provider、model）

2. 核心接口建议
   - IModelProvider.ChatAsync(ModelRequest)
   - IModelProvider.StreamAsync(ModelRequest)
   - IModelProvider.GetCapabilities()

3. 路由策略
   - 默认按 Agent 配置模型
   - 失败自动回退到备用模型
   - 记录调用延迟、错误率、token 用量

4. 配置策略
   - 支持多 Provider 并存
   - Key 与敏感配置仅通过环境变量或密钥管理注入
   - 每个 Agent 可独立指定主模型与回退模型

## 8. Definition of Done（DoD）

每个功能点完成必须满足：

1. API 文档（Swagger）可见
2. WebUI 可操作
3. 有审计日志
4. 有失败重试或错误提示
5. 有最小化测试（单测或集成测试）
6. 渠道消息收发链路有端到端测试
7. OpenAI / Claude 适配器通过统一契约测试
8. MCP 工具注册/执行链路有集成测试
9. 痛觉预警与拦截规则有单元测试
10. RAG 检索结果可通过最小效果基线测试
